# 12 — 性能分析：把知识用起来

---

## 这一章回答什么问题？

> `top` / `perf` / `strace` / `pprof` 输出的数字到底是什么意思？怎么根据这些数字找到瓶颈？

---

## 第一性原理

```text
性能分析的本质：
找到系统资源（CPU/内存/IO/网络）中拖后腿的那个。

关键：你得先理解操作系统是怎么管理这些资源的，
     才能看懂性能数据在说什么。
```

---

## 推导过程

### 性能分析的"USE 方法论"

```text
U — Utilization (利用率)
S — Saturation (饱和度)
E — Errors (错误数)

对每种资源检查：
CPU    → 利用率？就绪队列长度？上下文切换？
Memory → 使用量？Swap？OOM？
Disk   → IOPS？吞吐量？等待时间？
Network→ 吞吐量？丢包？重传？
```

---

## 指标速查表

| 指标 | 命令 | 正常 | 告警 |
|------|------|------|------|
| CPU 使用率 | `top` / `mpstat` | < 70% | > 90% |
| Load Average | `uptime` | < CPU 核数 | > CPU 核数 × 2 |
| 上下文切换 | `vmstat` (cs) | < 100k/s | > 500k/s |
| 内存使用 | `free -h` | < 80% | > 90% |
| Swap | `free -h` | 0 | > 0 (说明内存不够了) |
| 磁盘 IO 等待 | `iostat` (await) | < 10ms | > 50ms |
| 磁盘利用率 | `iostat` (%util) | < 70% | > 90% |
| 网络重传 | `ss -ti` | 0 | > 1% |

## 场景 1：CPU 接近 100%

```bash
# 1. 确认是哪个进程
top

# 2. 确认是用户态还是内核态
top  → 看 us (user) vs sy (system)

# 3. 如果是用户态高 → 程序在计算
#    如果是内核态高 → 可能是大量系统调用或上下文切换

# 4. 检查上下文切换
vmstat 1
pidstat -w -p <PID> 1

# 5. Go 程序：看 CPU profile
go tool pprof http://localhost:6060/debug/pprof/profile

# 6. 用 perf 看热点函数
perf top -p <PID>
```

---

## 场景 2：内存持续增长（疑似泄漏）

```bash
# 1. 看整体
free -h

# 2. 看进程内存
ps aux --sort=-%mem | head -10
cat /proc/<PID>/status | grep -i vm

# 3. 注意区分 VIRT 和 RES！
#    VIRT 很大但 RES 很小 → 正常（虚拟地址空间预分配）
#    RES 持续增长 → 可能是泄漏

# 4. Go 程序：看 heap profile
go tool pprof http://localhost:6060/debug/pprof/heap
# (pprof) top
# (pprof) list <function>

# 5. 看 goroutine 是否泄漏
go tool pprof http://localhost:6060/debug/pprof/goroutine
```

## 场景 3：响应变慢，CPU 不高

```bash
# 可能原因：IO 等待、锁竞争、网络延迟

# 1. 检查 IO 等待
iostat -x 1
# 看 await（平均等待时间）和 %util

# 2. 检查是否有大量 iowait
top → wa 列

# 3. 用 strace 看卡在哪个系统调用
strace -p <PID> -T   # -T 显示每个调用的耗时
strace -p <PID> -c   # 统计总结

# 4. Go 程序：看 block/mutex profile
# block: 阻塞在 channel、锁、网络 IO 的时间
go tool pprof http://localhost:6060/debug/pprof/block

# 5. 用 trace 看完整的调度过程
curl http://localhost:6060/debug/pprof/trace?seconds=5 > trace.out
go tool trace trace.out
```

---

## 场景 4：Load Average 很高但 CPU 不高

```text
Load Average = 就绪队列 + 不可中断的进程数 (D 状态)

CPU 不高但 Load 高：
→ 大量进程在等磁盘 IO！(D 状态)
```

```bash
# 1. 看 D 状态进程
ps aux | awk '$8 ~ /D/'

# 2. 看磁盘 IO
iostat -x 1

# 3. 看 iowait
top → wa 列

# 4. 定位哪个进程在等 IO
pidstat -d 1
```

## 场景 5：Go 程序的 GC 调优

```bash
# 开启 GC 追踪
GODEBUG=gctrace=1 ./myapp

# 输出解读：
# gc 1 @0.032s 2%: 0.10+0.26+0.012 ms clock,
#              0.10+0.10/0.20/0+0.036 ms cpu,
#              4->4->0 MB, 5 MB goal, 8 P

# 4->4->0 MB: GC 开始前堆 → GC 结束后堆 → 存活对象
# 如果"结束后"持续增长 → 内存泄漏
```

```go
import "runtime/debug"

// 关闭 GC（仅限特殊场景，如 benchmark）
debug.SetGCPercent(-1)

// 手动触发 GC
runtime.GC()

// 限制并发 GC 的 CPU 使用
debug.SetMaxThreads(10)
```

## 核心概念

| 概念 | 查什么命令 | 含义 |
|------|-----------|------|
| **us / sy / wa** | `top` | 用户态 / 内核态 / IO 等待 CPU 占比 |
| **Load Average** | `uptime` | 1/5/15 分钟平均就绪队列长度 |
| **cs** | `vmstat` | 每秒上下文切换次数 |
| **%iowait** | `top` | CPU 等待 IO 的时间占比 |
| **await** | `iostat -x` | 磁盘平均 IO 响应时间 |
| **RES / VIRT** | `top` | 常驻物理内存 / 虚拟内存 |
| **min_flt / maj_flt** | `ps` | 轻微/严重 缺页中断 |

## Go pprof 常用命令速查

```bash
# CPU profile (采样 30 秒)
go tool pprof http://localhost:6060/debug/pprof/profile?seconds=30

# Heap profile
go tool pprof http://localhost:6060/debug/pprof/heap

# Goroutine profile
go tool pprof http://localhost:6060/debug/pprof/goroutine

# Block profile (需要先开启)
# runtime.SetBlockProfileRate(1)
go tool pprof http://localhost:6060/debug/pprof/block

# Mutex profile (需要先开启)
# runtime.SetMutexProfileFraction(1)
go tool pprof http://localhost:6060/debug/pprof/mutex

# 在 pprof shell 中：
# (pprof) top         查看热点
# (pprof) list func   查看函数源码级分析
# (pprof) web         生成调用图（需要 graphviz）
# (pprof) png         生成 PNG 图片
```

## 实战：综合排查示例

```text
用户反馈："API 偶尔超时"
      │
      ▼
Step 1: top → CPU 30%, Mem 60%, 正常
      │
Step 2: vmstat 1 → cs 500000/s！上下文切换异常高
      │
Step 3: pidstat -w -p <PID> → nvcswch 极高
      │
Step 4: strace -p <PID> -c → futex 调用占 90%
      │
Step 5: 结论：锁竞争严重
      │
Step 6: Go pprof mutex profile → 找到热点锁
      │
Step 7: 修复 → 减少临界区范围 / 用 sync.Pool / 无锁结构
```

---

## 总结

> 性能分析的本质是用操作系统知识解读数字：CPU 高→看上下文切换、IO 等待；内存涨→看 VIRT vs RES、GC；Load 高 CPU 低→看 D 状态进程。`strace` + `perf` + `pprof` 三件套覆盖 90% 的场景。
