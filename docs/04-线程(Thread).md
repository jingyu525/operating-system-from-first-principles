# 04 — 线程 (Thread)

---

## 这一章回答什么问题？

> 有了进程为什么还需要线程？线程到底轻在哪里？

---

## 第一性原理

回顾进程：

```text
进程 = 程序 + 资源（地址空间、文件、信号等）
```

**问题来了：**

如果你写一个 Web 服务器，同时处理 10000 个请求：

- 创建 10000 个进程 → 每个都有完整的地址空间 → 内存爆炸
- 进程间通信 → 很麻烦（管道、共享内存、消息队列）

**你真正需要的是：**

> 共享地址空间，但各自独立执行的多个"执行流"。

这就是线程。

---

## 推导过程

```
进程模型：                线程模型：
┌─────────────┐          ┌─────────────────────┐
│   Process    │          │      Process         │
│ ┌─────────┐ │          │ ┌──────┬──────┬────┐ │
│ │ 地址空间  │ │          │ │Thread│Thread│... │ │
│ │ 文件描述符│ │          │ │  1   │  2   │    │ │
│ │ 信号处理  │ │          │ └──────┴──────┴────┘ │
│ │ ...       │ │          │   共享地址空间        │
│ │           │ │          │   共享文件描述符      │
│ │ 只一个执行流│ │          │   共享信号处理        │
│ └─────────┘ │          └─────────────────────┘
└─────────────┘
```

关键区别：

| | 进程 | 线程 |
|------|------|------|
| 地址空间 | 独立 | 共享 |
| 文件描述符 | 独立 | 共享 |
| 创建开销 | 大（复制地址空间） | 小（只分配栈） |
| 切换开销 | 大（切换地址空间） | 小（同地址空间） |
| 通信方式 | IPC | 共享内存 |
| 隔离性 | 强 | 弱（一个线程崩溃可能拖垮进程） |

## 内核线程 vs 用户线程

```
内核线程（Linux 中的线程）：
┌──────────┐ ┌──────────┐ ┌──────────┐
│ Thread 1 │ │ Thread 2 │ │ Thread 3 │
└────┬─────┘ └────┬─────┘ └────┬─────┘
     │            │            │
┌────┴────────────┴────────────┴────┐
│          Linux Kernel             │
│       (调度单位：task)             │
└───────────────────────────────────┘

用户线程（Go 的 goroutine）：
┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐
│  G1  │ │  G2  │ │  G3  │ │  G4  │  ← goroutine (用户态)
└──┬───┘ └──┬───┘ └──┬───┘ └──┬───┘
   │        │        │        │
┌──┴────────┴────────┴────────┴────┐
│         Go Runtime               │
│         GMP Scheduler             │
└────────────────┬─────────────────┘
                 │
┌────────────────┴─────────────────┐
│            OS Threads            │
└──────────────────────────────────┘
```

## 核心概念

| 概念 | 本质 |
|------|------|
| **线程** | 进程内的独立执行流，共享地址空间 |
| **TCB** | Thread Control Block（Linux 中线程和进程用同一结构 `task_struct`） |
| **TLS** | Thread Local Storage，线程私有数据 |
| **轻量级进程 (LWP)** | Linux 中线程的实现方式 |
| **用户态线程** | 不需要内核参与的线程（coroutine、goroutine） |

## Linux 是怎么实现的？

Linux 用 `clone()` 系统调用创建线程：

```c
// 创建进程：fork → clone(SIGCHLD, 0)
// 创建线程：clone(CLONE_VM | CLONE_FS | CLONE_FILES | CLONE_SIGHAND, 0)

// CLONE_VM：共享地址空间 ← 这是线程和进程的核心区别！
```

所以在 Linux 中，**线程本质上是一个共享更多资源的进程**。调度器看到的是 `task_struct`，不管它是不是线程。

```bash
# Linux 中查看线程
ps -eLf        # -L 显示线程
top -H         # -H 显示线程视图
ls /proc/self/task/  # 每个线程一个子目录
```

## Go 是怎么利用它的？

Go 的并发模型建立在 OS Thread 之上：

```text
GMP 模型：
G (Goroutine) → 用户态轻量线程，栈 ~2KB 起步
P (Processor) → 逻辑处理器，管理 G 的本地队列
M (Machine)   → OS Thread，真正执行代码
```

```text
 ┌───┐ ┌───┐ ┌───┐ ┌───┐
 │ G │ │ G │ │ G │ │ G │  ← goroutine 队列
 └─┬─┘ └─┬─┘ └─┬─┘ └─┬─┘
   │     │     │     │
   └──┬──┴──┬──┴──┬──┘
      │     │     │
   ┌──┴──┐ ┌┴───┐ ┌┴───┐
   │  P  │ │ P  │ │ P  │  ← 逻辑处理器
   └──┬──┘ └─┬──┘ └─┬──┘
      │      │      │
   ┌──┴──────┴──────┴──┐
   │     OS Threads    │  ← M (Machine)
   └─────────┬─────────┘
             │
        Linux Scheduler
             │
            CPU
```

**为什么 goroutine 比线程轻？**

| | OS Thread | Goroutine |
|------|-----------|-----------|
| 创建者 | OS 内核 | Go Runtime |
| 栈大小 | ~8MB（固定） | ~2KB（可增长） |
| 切换者 | OS 调度器 | Go 调度器（用户态） |
| 切换开销 | ~1-10μs（保存/恢复所有寄存器+切换地址空间） | ~200ns（在用户态完成） |
| 数量上限 | ~几千（受内存和调度限制） | ~百万级 |

### 什么是"陷入内核"？

CPU 有两个权限级别：

```text
┌───────────────────────────────┐
│         内核态 (Ring 0)        │  ← 管理员权限
│   可以做任何事：               │
│   - 直接操作硬件（磁盘、网卡）  │
│   - 修改页表                  │
│   - 切换进程                  │
├───────────────────────────────┤
│         用户态 (Ring 3)        │  ← 普通权限
│   只能做安全的事：             │
│   - 加减乘除                  │
│   - 读写自己的内存             │
│   - 调用函数                  │
│   ❌ 不能切进程                │
│   ❌ 不能碰硬盘                │
└───────────────────────────────┘
```

你的 Go 程序跑在**用户态**，内核跑在**内核态**。当你需要内核帮忙时（比如切换 OS Thread），必须穿过这道墙——**这就是"陷入"**。

"陷入"本质上是 CPU 执行了 `SYSCALL` 指令：

```text
用户态                         内核态
  │                              │
  │  "内核大哥，帮我切线程！"      │
  │                              │
  │──── SYSCALL 指令 ───────────→│  ← 这就是"陷入"
  │   (CPU 切到 Ring 0)          │
  │                              │
  │                    内核执行切换：
  │                    保存所有寄存器
  │                    切换页表 (CR3)
  │                    恢复新线程的寄存器
  │                              │
  │←─── SYSRET 指令 ────────────│  ← 回到用户态
  │                              │
```

一次"陷入-返回"的不可见代价：

```text
SYSCALL 指令本身：       ~100ns
保存/恢复寄存器：         ~几十 ns
切换栈（用户栈→内核栈）：  ~几十 ns
权限检查：               ~几十 ns
可能刷 TLB：            ~几百 ns

总共：~500ns - 1μs
```

**goroutine 为什么不需要陷入？** 因为 goroutine 的切换全部在 Go Runtime 里完成，不经过内核：

```text
OS Thread 切换：
  用户态 → [SYSCALL 陷入] → 内核态 → 内核调度器 → 切线程 → [SYSRET] → 用户态
  代价：~1-10μs（出门一趟，要过安检）

goroutine 切换：
  用户态 → Go Runtime 调度器（保存几个寄存器 → 换栈 → 恢复）→ 用户态
  代价：~200ns（从客厅走到卧室，全程不出家门）
```

**类比：**

```text
OS Thread 切换 = 你在家 → 出门 → 坐车 → 去市政府办事大厅 → 办完 → 坐车 → 回家
               贵，因为要"出趟远门"

goroutine 切换 = 你在家 → 从客厅走到卧室
                便宜，因为根本没出门
```

"陷入"就是**出家门去市政府**的过程，goroutine 切换全程**不出家门**。

## 常见面试题

**Q: 进程和线程的区别？**

A: 核心区别是**地址空间是否共享**。进程之间不共享地址空间（隔离性强），线程之间共享地址空间（轻量）。这决定了创建开销、切换开销、通信方式的差异。

**Q: goroutine 是线程吗？**

A: 不是。goroutine 是 Go Runtime 管理的用户态协程，多个 goroutine 运行在少量 OS Thread 上。goroutine 切换不需要陷入内核，开销极小。

**Q: 为什么线程太多反而性能下降？**

A: 三个原因：
1. 每个线程的栈占内存（~8MB），10000 个线程 = 80GB 虚拟内存
2. 上下文切换开销线性增长
3. 调度器本身的开销

## 实战

```bash
# 查看进程的线程数
cat /proc/$(pidof myapp)/status | grep Threads

# 查看每个线程的状态
ps -T -p $(pidof myapp)

# 用 perf 查看上下文切换
perf stat -e context-switches myapp
```

```go
// labs/thread/main.go
package main

import (
    "fmt"
    "runtime"
    "sync"
)

func main() {
    fmt.Printf("CPU 核数: %d\n", runtime.NumCPU())
    fmt.Printf("当前 goroutine 数: %d\n", runtime.NumGoroutine())
    fmt.Printf("OS Thread 数: %d\n", runtime.GOMAXPROCS(0))

    var wg sync.WaitGroup
    for i := 0; i < 10; i++ {
        wg.Add(1)
        go func(id int) {
            defer wg.Done()
            fmt.Printf("goroutine %d running\n", id)
        }(i)
    }
    wg.Wait()
}
```

## 总结

> 线程是进程内共享地址空间的独立执行流。OS Thread 重但通用，goroutine 轻但需要 Runtime 支持。
